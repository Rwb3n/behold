# Taste

Assess whether something feels right — whether it belongs, whether it coheres with the surrounding design, whether it would make the creator wince or nod. Taste is pattern recognition trained on accumulated exposure to good and bad work. It is the faculty that notices "this works but something is off."

**Framework:** If you have a codified taste document (style guide, design principles), read it before evaluating. Evaluate against documented heuristics, not just gut feel.

## When to use
- Evaluating naming choices, API shapes, or interface designs
- Deciding whether a solution is elegant or merely correct
- Checking whether new work fits the existing style and philosophy

## Guiding questions
1. Does this feel like it belongs here, or does it feel bolted on?
2. Would someone familiar with this codebase/system recognize this as consistent?
3. Is there a simpler way to express the same intent?
4. Does the naming reveal intent, or does it require explanation?
5. In six months, will this choice still feel right, or will it feel like a shortcut?

## Output structure
- Coherence verdict (belongs / friction / foreign)
- What feels right and should be preserved
- What feels off and why (even if technically correct)
- Suggested alternative if friction detected
