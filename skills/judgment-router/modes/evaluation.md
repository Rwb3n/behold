# Evaluation

Assess whether an output meets its quality bar. Apply standards appropriate to the context — production code demands more than a prototype, a design doc more than a scratch note. The goal is a clear accept/revise/reject verdict.

## When to use
- Reviewing code, text, or designs before they ship
- Deciding whether a draft is ready for the next stage
- Comparing actual output against stated requirements or intent

## Guiding questions
1. What is the quality bar for this context? (prototype vs production, internal vs external)
2. Does the output fulfill the stated intent — not just technically, but in spirit?
3. What are the weakest parts? Would they cause problems downstream?
4. Is there anything missing that should be present, or present that should be absent?
5. If I saw this for the first time with no context, would it make sense and feel complete?

## Output structure
- Verdict (accept / revise / reject)
- Strengths worth preserving
- Weaknesses ranked by severity
- Specific revision guidance (if revise)
